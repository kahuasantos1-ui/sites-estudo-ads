🏃 Manifesto Ágil & Scrum cai na Q1, Q2, Q11

Os 4 valores do Manifesto Ágil (2001)

O manifesto não diz que o lado direito não tem valor — diz que o lado esquerdo vale MAIS:

Valorizamos MAIS......do que
Indivíduos e interaçõesProcessos e ferramentas
Software funcionandoDocumentação abrangente
Colaboração com o clienteNegociação de contratos
Responder a mudançasSeguir um plano
⚠️ Pegadinha clássica: as alternativas erradas invertem a ordem ("processos mais que pessoas", "documentação mais que software"). Leia com calma!

Papéis do Scrum

PapelResponsabilidade
Product Owner (P.O.)Maximiza o valor do produto, gerencia o Product Backlog
Scrum MasterRemove impedimentos, facilita o processo e garante a correta aplicação do framework
Time de DesenvolvimentoProfissionais multidisciplinares que entregam os incrementos

Eventos

  • Sprint: ciclo fixo de 2–4 semanas onde o trabalho é realizado
  • Sprint Planning: planejamento do que será feito
  • Daily: reunião diária de 15 minutos de alinhamento
  • Sprint Review: demonstração do que foi entregue
  • Sprint Retrospective: reflexão e melhoria contínua do processo
🎯 Direto pra prova: ao final de cada Sprint deve ocorrer uma entrega incremental potencialmente utilizável do produto — não é só documentação, não é o fim do projeto, não é deploy obrigatório.
📋 Kanban & WIP cai na Q3

Lema: "Pare de começar, comece a terminar." Origem no Sistema Toyota, adaptado pra software por David J. Anderson.

Os 3 princípios

  • Visualizar o fluxo: o quadro com colunas (Backlog → Dev → Teste → Done) torna o trabalho invisível em cartões visíveis
  • Limitar o WIP (Work in Progress): número máximo de tarefas por coluna — ex.: DEV [3]
  • Gerenciar o fluxo: identificar gargalos (coluna que acumula) e agir coletivamente
🎯 Direto pra prova: limitar o WIP serve pra melhorar o fluxo de trabalho e reduzir gargalos. Multitarefa gera custo de troca de contexto e atrasa tudo.

Métricas de tempo

MétricaDefinição
Lead TimeTempo total — do pedido do cliente até a entrega
Cycle TimeTempo de execução — de quando o trabalho começou até terminar

Sistema puxado: o profissional só inicia tarefa nova se tiver capacidade livre (espaço no WIP) — ao contrário do sistema empurrado tradicional.

🗂️ Git & Versionamento cai na Q9, Q13

A Gerência de Configuração (GCS) garante que o sistema seja reprodutível em qualquer momento. O principal benefício do Git: controle e rastreamento de alterações no código — quem mudou, o quê, quando e por quê.

As 4 áreas de trabalho

ÁreaO que é
Working DirectorySeus arquivos "vivos" no sistema
Staging AreaO "limbo" — você prepara o que vai subir
Repo LocalHistórico (SHA-1) no seu PC
RemoteRepositório central (GitHub/GitLab) pra colaboração

Comandos essenciais

ComandoO que faz
git cloneCopia um repositório remoto pro seu PC
git pullBaixa alterações do remoto
git pushEnvia alterações locais pro remoto ⭐
git initCria um repositório novo
git statusMostra o estado dos arquivos
💡 Macete: push = empurrar (manda pro servidor) · pull = puxar (traz do servidor). Enviar alterações locais pro remoto = git push.

Fluxo de trabalho: pull → checkout -b feature → add → commit → push → Pull Request (revisão antes de entrar na main).

🐳 Docker & Containers cai na Q6, Q12, Q16

Docker empacota a aplicação com todas as dependências num container que roda igual em qualquer lugar. É a cura do famoso "funciona na minha máquina". 🩹

Imagem vs Container

ConceitoO que éAnalogia
ImagemModelo (template) para criação de containers — imutávelA receita do bolo 📋
ContainerInstância em execução criada a partir da imagemO bolo assado 🎂

Container vs Máquina Virtual

  • Containers são mais leves que VMs — compartilham o kernel do S.O. hospedeiro, não emulam hardware
  • Docker não elimina a necessidade de sistema operacional (usa o kernel do host!)
  • Docker roda em Linux, Windows e Mac — não é "apenas Windows"
  • Containers não exigem hardware dedicado
🎯 Direto pra prova: empresa quer que a aplicação funcione igual em desenvolvimento, homologação e produção? Containers Docker.
Empresa sofre com "funciona na minha máquina", conflito de dependências e diferenças entre ambientes? Docker e padronização de ambientes.
☸️ Kubernetes — Orquestração cai na Q7, Q14, Q15, Q17

O Kubernetes (k8s) é a plataforma para gerenciar containers em larga escala — o "maestro" da orquestra. Automatiza implantação, dimensionamento e gestão.

Componentes

ComponentePapel
ClusterConjunto de máquinas (nodes)
Control PlaneO "cérebro" — decide onde os containers rodam
Nodes workersMáquinas que executam o trabalho
PodA menor unidade — você sobe pods, não containers soltos

A "mágica" da autonomia

RecursoO que faz
Auto-scalingAumenta ou reduz containers conforme a demanda (monitora CPU e reage sozinho)
Self-healingRecuperação automática de containers com falha — reinicia em servidor saudável
Rolling updateAtualiza versão por versão, sem downtime
📌 Exemplo Black Friday: tráfego sobe 100% à meia-noite → k8s vê CPU a 90% → escala de 10 pra 100 pods sozinho → de manhã o tráfego cai → k8s desliga os extras. 🛒
🎯 Direto pra prova: startup com crescimento acelerado que precisa de escalabilidade + alta disponibilidade + recuperação automática de falhas = Kubernetes. (GitHub versiona, HTML estrutura página, Scrum Board organiza tarefas, SQL Server guarda dados — nenhum escala containers!)
🔄 CI/CD, DevOps & Testes cai na Q4, Q5, Q8, Q10, Q18, Q19

CI — Integração Contínua

Função principal: integrar código frequentemente com testes automatizados. Cada commit dispara build + testes — o erro aparece em minutos, não em semanas.

CD — Entrega vs Implantação Contínua

ConceitoComo funciona
Continuous DeliveryCódigo sempre pronto pra produção, mas o deploy final tem aprovação manual
Continuous DeploymentO deploy ocorre automaticamente após validações automatizadas — zero intervenção humana

Pipeline CI/CD

  • Busca principalmente: automatizar integração, testes e entrega de software
  • Pipeline madura reduz principalmente: erros humanos e tempo de entrega

DevOps

Cultura que integra Development + Operations. A integração busca: melhorar colaboração, automação e entrega contínua — nunca separar equipes nem aumentar burocracia.

Testes

TipoObjetivo
UnitárioTesta um módulo/função isoladamente
IntegraçãoValidar a comunicação entre componentes integrados ⭐
💡 Macete: unitário = peça sozinha · integração = peças conversando. Se a questão fala em "comunicação entre módulos/componentes" = teste de integração.

🏆 Desafio dissertativo

A questão aberta da revisão. Tente responder de cabeça antes de abrir o gabarito.

Uma empresa deseja modernizar seu processo de desenvolvimento de software utilizando:

  • Scrum
  • Git
  • Docker
  • Kubernetes
  • CI/CD

Explique como cada tecnologia/metodologia pode contribuir para: qualidade de software, velocidade de entrega, escalabilidade, colaboração entre equipes e redução de falhas em produção.

✍️ Resposta-modelo

A modernização funciona porque cada peça resolve uma dor específica do processo:

🏃 Scrum
  • Colaboração: papéis claros (P.O., Scrum Master, time) e eventos como Daily e Review mantêm todos alinhados com o cliente.
  • Velocidade: sprints curtas geram entregas incrementais frequentes em vez de um "big bang" no fim do projeto.
  • Qualidade: feedback contínuo dos stakeholders e retrospectivas corrigem o rumo cedo.
🗂️ Git
  • Qualidade: todo código passa por revisão via Pull Request antes de entrar na main.
  • Colaboração: branches permitem que vários devs trabalhem em paralelo sem se atropelar.
  • Redução de falhas: histórico completo e rastreável — qualquer versão pode ser restaurada (rollback).
🐳 Docker
  • Redução de falhas: ambientes padronizados eliminam o "funciona na minha máquina" — dev, homologação e produção idênticos.
  • Velocidade: a aplicação já vai empacotada com todas as dependências, pronta pra rodar em qualquer lugar.
☸️ Kubernetes
  • Escalabilidade: auto-scaling aumenta ou reduz containers automaticamente conforme a demanda.
  • Redução de falhas em produção: self-healing reinicia containers com falha sozinho; rolling updates atualizam sem downtime — alta disponibilidade.
🔄 CI/CD
  • Velocidade de entrega: integração, testes e deploy automatizados — código chega em produção em minutos.
  • Qualidade: testes automatizados rodam a cada commit, pegando bugs cedo.
  • Redução de falhas: menos passos manuais = menos erro humano.

Conclusão: Scrum organiza como o time trabalha, Git protege o código, Docker padroniza os ambientes, Kubernetes garante escala e resiliência e o CI/CD automatiza o caminho até produção. Juntos formam a base da cultura DevOps: colaboração, automação e entrega contínua.
